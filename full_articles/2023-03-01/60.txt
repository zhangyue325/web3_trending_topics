构建协议没有唯一正确的方法，只要你选择的方法包括构建协议的需求。然而，根据协议类型的不同，通常有一种最适合的上市 (GTM) 策略。如今，协议构建需求的方式通常由两种 GTM 方法决定：

1. 网络驱动的 GTM 方法
2. 产品主导的 GTM 方法
让我们简要地讨论这两种策略，并检查每种策略何时最有效。

网络驱动协议 GTM
----------

网络驱动协议 GTM 专注于构建具有广泛分布和无许可访问的协议，然后鼓励其他人在网络之上进行构建。

这种方法通常通过以下方式鼓励采用：

* 无许可访问 ( 例如，以太坊 )
* 强大的业务开发能力 ( 例如，Polygon)
* 代币标准或合约集成 ( 例如，Metaplex)
* 资源市场 ( 例如，Akash， Livepeer)
* 应用程序集成和强大的文档 ( 例如，XMPT， Reservoir， Lens)
需要明确的是，这一策略并不仅仅意味着「建立一个协议，人们就会来。剧透警告：他们不会。如果你不帮助培养需求，那么你将只剩下优秀的代码而没有用户。相反，网络驱动的方法必须与 GTM 战略相结合，以激发最初的吸引力，无论是通过资助潜在的应用程序，促进开发人员活动，与设计合作伙伴 / 旗舰客户合作，还是以其他方式形成伙伴关系和计划，为你的协议带来需求。采用这种方法可以让团队专注于协议，并有助于生成初始团队可能没有预料到的用户活动。

这种方法的另一个核心优势是能够保持可信的中立。随着时间的推移，应用程序和协议可能会有不同的利益，这可能很难说服企业家在一个平台上进行开发。如果企业家认为你的协议会与他们竞争，或者像 Twitter 那样通过关闭其客户端竞争对手的 API 访问来施加控制，他们就会选择更中立的平台。

以产品为导向的协议 GTM
-------------

以产品为导向的协议 GTM 专注于在协议之上构建单个应用程序、工具或产品，以快速启动需求。

这种 GTM 策略在以下方面最有效：

* 初始工具套件或产品 ( 例如，Mirror)
* 增强可发现性非常重要的协议 ( 例如，OpenSea + Seaport)
* 一个无状态协议 ( 例如，一个合约库 )，这样应用程序对于价值捕获是必要的 ( 例如，支付分割合约 )
* 一些即时通讯应用 ( 如 Dialect)
* 具有特定焦点的核心原语 ( 例如 Uniswap)
* 核心网络基础设施的消费者分布 ( 即 Hivemapper、DIMO)
* 基于市场的协议 ( 例如，带有音乐 NFTs 的声音和目录 )
* 在控制独特内容和体验非常重要的类别中 ( 例如 Oncyber)
产品主导的协议开发和 GTM 策略的目标是：充当协议活动的初始楔子和首要驱动力。构建人们在你的协议上使用的核心应用程序有很多好处，包括 Dogfooding 协议、推动初始协议牵引力，以及可能在应用程序层实现协议获利。

Dogfooding 是指你在协议上构建一个应用程序，然后你的团队可以使用和测试。在这样做的过程中，发现哪些有效，哪些需要更多工作，就像用户可能做的那样。最初，ENS 的使用非常具有挑战性，需要用户参与荷兰式拍卖过程来购买他们的用户名。最终，该团队构建了一个易于使用的界面，使预订和购买 ENS 域名变得更加简单。类似地，Uniswap 构建了其核心应用程序，至今仍在广泛使用，尽管现在有许多其他应用程序也利用了该协议的流动性。

其次，当一个协议的竞争优势来自于吸引用户的注意力时，以产品为导向的协议方法很重要。例如，NFT( 如 Seaport) 的流动性并不锁定在资产层面，因此市场目前基于产品体验 ( 如 Blur) 或可发现性 ( 如 ENS Vision) 争夺用户。

最后，无状态的协议 ( 即容易分叉，因此可能失去其竞争优势 ) 可以提供很大的实用性，但仍然需要附加到协议的应用程序来获取价值。这些无状态协议的运作方式通常与传统的开源软件类似，在传统的开源软件中，大型企业通过高级功能或更强大的产品体验来实现开源软件的盈利。

如何在产品主导和网络驱动的 GTM 之间做出选择
------------------------

无论你采用何种方法，产生对协议的需求都是成功的关键。创造需求的协议——无论是通过构建自己的产品还是通过激励活动——将在竞争中胜过无法找到构建需求方法的协议。

据观察，可推广的协议——提供广泛用例的协议——通常受益于保持中立或促进广泛应用的程序。相反，特定的协议——具有目标用户角色的协议——通常受益于利用产品主导的协议 GTM 方法构建接口或应用程序。此外，需要引起用户关注以获取价值的协议也应该将重点放在与其协议一起构建自己的应用程序上。

遵循这个大纲，大多数协议通过采用产品主导的协议 GTM 方法将有更大的成功机会，因为它允许协议快速启动并拥有它们的初始需求。
